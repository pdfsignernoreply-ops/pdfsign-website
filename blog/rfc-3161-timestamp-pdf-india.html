RFC 3161 Timestamp in PDF Signing — What It Is and Why It Matters in India

By PDFSign  ·  Apr 2026  ·  8 min read
Home › Blog › RFC 3161 Timestamp in PDF Signing

Every time you sign a PDF with your DSC token, software like PDFSign can contact a Timestamp Authority (TSA) server and embed a cryptographic timestamp into the document. This timestamp is governed by an IETF standard called RFC 3161.

Why does this matter? Because the timestamp is the difference between a signature that is valid for 2 years (your DSC's lifetime) and one that is valid indefinitely. It is one of the most important — and most overlooked — aspects of digital signing for Indian government portals, tenders, and legal documents.

What Is RFC 3161?

RFC 3161 is an IETF (Internet Engineering Task Force) standard formally titled "Internet X.509 Public Key Infrastructure — Time-Stamp Protocol (TSP)". It was published in 2001 and defines a protocol for requesting and receiving trusted timestamps from a Timestamp Authority server.

In plain terms: RFC 3161 is the universally accepted method by which a trusted third-party server — the Timestamp Authority (TSA) — certifies that a specific piece of data (such as a signed PDF's hash) existed at a specific moment in time.

The TSA's timestamp token is itself digitally signed by the TSA's own certificate, creating a cryptographic chain: TSA trusts the time → TSA signs the hash → verifier trusts the TSA → verifier trusts the time.

How RFC 3161 Timestamping Works — Step by Step

1Document Is Signed

Your DSC token signs the PDF document. A cryptographic hash (SHA-256) of the signed content is computed. The private key never leaves the hardware token.

2Timestamp Request Sent to TSA

The signing software (PDFSign) sends the document hash — not the document itself — to a Timestamp Authority server over HTTPS. The document content is never transmitted.

3TSA Issues a Signed Timestamp Token

The TSA server records the current UTC time, combines it with the hash, and signs the resulting timestamp token using the TSA's own private key. The TSA's certificate chain is embedded in the token.

4Token Embedded Into PDF

The timestamp token is inserted into the PDF signature structure as an unsigned attribute. It is now permanently part of the signed document.

5Verification (Any Time, Offline)

A verifier (Acrobat Reader, any PAdES-compliant tool) reads the timestamp token, verifies the TSA's signature, and confirms the hash matches the document. This proves the document existed in its current form at the TSA-certified time.

What the Timestamp Token Contains

// RFC 3161 Timestamp Token (simplified structure) { version: 1, policy: "1.3.6.1.4.1.4146.2.1", // TSA's timestamping policy OID messageImprint: { hashAlgorithm: "SHA-256", hashedMessage: "a3f7c4..." // SHA-256 of signed PDF content }, serialNumber: 1234567890, genTime: "2026-04-12T14:32:07Z", // UTC time certified by TSA tsa: "GlobalSign Timestamping CA", signature: "[TSA's RSA-2048 signature]" // Proves token is authentic }

This token is then embedded inside the PDF's signature dictionary. It cannot be modified without breaking the TSA's signature — and if anyone tampers with the document after timestamping, the hash will no longer match, and the signature will show as invalid.

The Difference Between the Signer's Clock and a TSA Timestamp

Many signing tools embed the signing time using the local system clock of the signer's computer — not an RFC 3161 timestamp. This is a critical difference:

  • System clock time: set by the user; can be set to any date; not cryptographically verifiable; courts do not trust it as independent evidence
  • RFC 3161 TSA time: independently certified by a third-party authority; signed with the TSA's private key; cannot be forged; universally accepted by Adobe, courts, and portal operators
If your signing software shows "Signing time: [date]" from the computer's clock rather than from a TSA, the timestamp is not RFC 3161 compliant and will not protect your signature after the DSC certificate expires.

How to Verify a Timestamp in Adobe Acrobat Reader

1Open the Signed PDF

Open the PDF in Adobe Acrobat Reader (free download from Adobe).

2Open the Signature Panel

Click the signature field or open the left-side panel and select "Signatures".

3Click Signature Properties

Right-click the signature entry and choose "Show Signature Properties" or click the link at the bottom of the panel.

4Check Date/Time Source

Look at the "Date/Time" line. It should say "from the signature's timestamp" — not "from the clock on the signer's computer". The TSA issuer name (e.g., "GlobalSign", "DigiCert") should also be visible under timestamp details.

If the date/time source says "from timestamp", your signature is RFC 3161 compliant and will remain verifiable after the DSC certificate expires.

RFC 3161 and the PAdES Standard

RFC 3161 timestamps are a core component of the PAdES (PDF Advanced Electronic Signatures) standard:

  • PAdES-B: No timestamp — signature only
  • PAdES-T: Adds RFC 3161 timestamp (the "T" stands for Timestamp)
  • PAdES-LT: Adds OCSP/CRL data to enable offline verification
  • PAdES-LTA: Adds an archive timestamp for very long-term preservation

PDFSign produces PAdES-LT signatures — which include both the RFC 3161 timestamp and the OCSP revocation status data embedded in the PDF. This is the recommended level for Indian government, legal, and commercial use.

Privacy Note — What Does the TSA See?

The TSA server receives only a SHA-256 hash of the document's signed content. The actual document content is never transmitted to the TSA. The hash is a fixed-length fingerprint (64 hex characters) that proves a document existed but reveals nothing about its contents.

This means timestamping works for confidential documents — tender financial bids, legal agreements, bank statements — without exposing any content to the TSA.

Frequently Asked Questions

Does RFC 3161 timestamping cost extra?

No. PDFSign includes RFC 3161 timestamping in every plan, including the free trial. There is no additional fee per document or per timestamp. Some enterprise TSA providers charge for high-volume usage, but this is transparent to end users.

What if the TSA server is down when I try to sign?

PDFSign will attempt to reach the TSA and will notify you if the connection fails. In that case, you can retry — the document signing itself does not require internet (only the timestamp request does). If no timestamp can be obtained, PDFSign will warn you before completing the signature.

Is my signed PDF tamper-evident?

Yes. Any modification to the PDF after signing — including changing a single character, adding a page, or altering the signature field — will cause the cryptographic hash to change. The signature verification will immediately show as invalid. This is independent of the timestamp.

Can I add a timestamp to an already-signed PDF?

You can add a document-level timestamp to an already-signed PDF — this is what PAdES-LTA does. However, this does not change the original signature timestamp. The original signature's validity is determined by the timestamp that was (or was not) embedded at the time of signing.
