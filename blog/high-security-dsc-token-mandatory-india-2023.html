High Security DSC Tokens Mandatory in India — What Changed in 2023

By PDFSign  ·  Nov 2025  ·  6 min read
Home › Blog › High Security DSC Tokens Mandatory in India — What Changed in 2023

If you have applied for a new Class 3 Digital Signature Certificate (DSC) in 2023 or later, you received a USB token that looks slightly different from the older ePass 2003 or ProxyKey tokens that were issued before. The label might say "High Security" or show a FIPS certification mark. You may have noticed the serial number starts with "HS" or "WD" rather than the older format.

This is not a coincidence. The Controller of Certifying Authorities (CCA), operating under the Ministry of Electronics and Information Technology (MeitY) and empowered by the Information Technology Act, 2000, issued a directive in 2023 mandating that all new Class 3 DSC tokens must meet FIPS 140-2 Level 3 security certification. This article explains what changed, why it matters, which tokens qualify, and what it means for your existing setup.

Background: The CCA and How DSCs Are Governed in India

Digital Signature Certificates in India are governed by the Information Technology Act, 2000. Section 17 of the IT Act establishes the Controller of Certifying Authorities, a statutory authority that licenses and oversees Certifying Authorities (CAs). In India, licensed CAs include eMudhra, Capricorn CA, NSDL e-Governance Infrastructure Limited, SafeScrypt (a Sify company), (n)Code Solutions, and a few others.

The CCA sets the technical and procedural standards that CAs must follow when issuing DSCs. These standards cover key length, certificate validity period, revocation procedures, identity verification requirements, and — critically — the security specifications of the hardware token on which the private key is stored.

Before 2023, the token standard required FIPS 140-2 Level 2 certification. The 2023 directive raised this requirement to Level 3 for all new Class 3 DSC issuances.

What Changed in 2023: The FIPS 140-2 Level 3 Mandate

FIPS 140-2 (Federal Information Processing Standard 140-2) is a US government standard for cryptographic module security, published by NIST. It defines four security levels, with Level 1 being the minimum and Level 4 being the highest. The standard is widely adopted internationally as the benchmark for secure hardware cryptographic devices.

FIPS LevelKey RequirementDSC Token Status (India)
Level 1Basic software/hardware securityNot sufficient for Class 3 DSC
Level 2Tamper-evident physical securityRequired before 2023
Level 3Tamper-resistant, key zeroisation on attackMandatory from 2023 for new Class 3 DSC
Level 4Complete physical protection envelopeNot required for standard DSC

The core difference between Level 2 and Level 3 is what happens when the hardware is physically attacked. A Level 2 device must show evidence of tampering (tamper-evident seals). A Level 3 device must actively resist physical intrusion and automatically erase (zeroize) the private key if the secure boundary is breached. At Level 3, the private key generation must happen on-chip — the key is never in plaintext outside the secure hardware boundary.

Why the Change Was Made

The primary motivation was to prevent private key cloning. On older FIPS 140-2 Level 2 tokens, it was theoretically possible — with physical access to the token and sophisticated hardware attacks — to extract the private key from the token's memory and copy it to another device. A cloned private key allows someone else to sign documents as you without your knowledge or consent.

As India's digital economy has grown and the value of digitally signed documents — government tenders, company filings, income tax returns, property transactions — has increased, so has the incentive for malicious actors to clone DSC private keys. The 2023 mandate closes this vulnerability by requiring hardware that physically destroys the key rather than exposing it under attack.

Additional reasons for the upgrade include:

  • Alignment with international best practices for high-assurance digital identity tokens
  • Preventing key export: on a Level 3 token, the private key cannot be exported by software even by the token owner — it is permanently bound to the hardware
  • Meeting the increasing assurance requirements of portals like MCA21, GeM, NIC e-Procurement, Income Tax, and DGFT

Difference Between Old Tokens and High Security Tokens

Key Generation

On older Level 2 tokens, some models allowed the private key to be generated externally (on the PC) and then imported onto the token. This meant the key existed in computer memory — however briefly — and could potentially be intercepted. On a Level 3 High Security token, the key pair is generated inside the secure element on the token itself. The private key never exists outside the hardware. The CA software sends a key generation command to the token; the token generates the key internally and returns only the public key for certificate issuance.

Tamper Resistance vs. Tamper Evidence

Older Level 2 tokens were tamper-evident: physical stickers or seals would show if someone had opened the device. Level 3 tokens are tamper-resistant: the internal circuitry is designed so that probing the hardware or applying voltage attacks triggers automatic key erasure. The secure boundary is actively defended rather than merely labeled.

Key Export

On a Level 3 token, software cannot export the private key. The signing operation happens inside the token's secure element. The document hash is sent to the token, the token signs it internally with the private key, and the signature is returned — the private key itself never leaves the hardware under any circumstances.

How to Identify if Your Token Is High Security

There are several ways to check whether your USB DSC token meets the FIPS 140-2 Level 3 requirement:

Check the Serial Number

Serial number prefixes are the quickest identifier:

  • HYP2003 tokens: High Security tokens have serial numbers beginning with HS (e.g., HS2XXXXXXX). Older non-HS tokens have serial numbers in other formats. If your HYP2003 serial starts with HS, it is a High Security FIPS 140-2 Level 3 token.
  • Proxkey (Watchdata) tokens: High Security tokens have serial numbers beginning with WD (e.g., WD3XXXXXXX). These are the Proxkey WD series, which are FIPS 140-2 Level 3 certified.

Check the Token Body or Packaging

Many High Security tokens print "FIPS 140-2 Level 3" or "High Security" directly on the USB device body or on the packaging/card it was delivered with. Look for this text on the physical token.

Check Through Token Management Software

Open your token management application (SafeNet Authentication Client, ePass2003 Manager, or the Proxkey Manager). Navigate to the token information or properties section. High Security tokens typically report their FIPS certification level in the token details screen.

Check the DSC Certificate Itself

Open the certificate on your token (through the Windows Certificate Store or the token management software). Some CAs include a policy OID or a certificate policy statement that specifies the token security level. Your CA's certificate policy document (available on the CA's website) will list which tokens they use for different certificate classes.

Quick check: If your HYP2003 token serial number begins with HS, or your Proxkey serial number begins with WD, you have a FIPS 140-2 Level 3 High Security token and you are compliant with the 2023 CCA mandate.

Certifying Authorities Issuing High Security Tokens

All licensed CAs in India are required to issue new Class 3 DSCs only on FIPS 140-2 Level 3 tokens from 2023. The main CAs and the tokens they commonly issue include:

What Happens to Existing Old Tokens?

The 2023 mandate applies only to new issuances. If you received your Class 3 DSC before the mandate on an older Level 2 token, your DSC continues to be legally valid until its expiry date. Government portals — GeM, CPPP, NIC e-Procurement, MCA21, Income Tax, DGFT — will continue to accept signatures from certificates on pre-mandate tokens as long as the certificate has not expired and has not been revoked.

The change takes effect at renewal. When your DSC expires and you apply for a new one, the CA will issue it on a FIPS 140-2 Level 3 High Security token. In some cases, if your existing token is already HS certified, the CA may load the new certificate onto the same hardware. In other cases, you will receive a new physical token.

Do not delay DSC renewal. If your DSC expires and you have not renewed it, you will not be able to sign documents until the new DSC is issued and the token arrives. Renewal typically takes 1–3 business days, but plan ahead especially before major tender deadlines.

Impact on PDF Signing Software

From the perspective of PDF signing software, High Security tokens behave identically to their predecessors. They communicate via the same PKCS#11 interface standard. The driver (middleware) handles the low-level communication between the software and the token's secure element.

PDFSign uses the PKCS#11 standard to communicate with all USB DSC tokens. Because the interface is standardised, PDFSign works with High Security tokens without any special configuration:

  • HYP2003 HS series: Use the same ePass2003 middleware driver as before. PDFSign detects the certificate automatically.
  • Proxkey WD series: Use the Proxkey middleware. PDFSign detects the certificate automatically.
  • ePass 3003: Use the ePass2003 or ePass3003 middleware. No change in workflow.
  • Feitian and WD Pro tokens: Use the vendor's PKCS#11 middleware. PDFSign supports all PKCS#11-compliant tokens.

The signing workflow in PDFSign is identical regardless of whether you are using an older Level 2 token or a new High Security Level 3 token. The software sends the document hash to the token, the token signs it inside its secure element, and the signature is embedded in the PDF. No changes to your PDFSign configuration are needed when you receive a new HS token.

How to Check If Your New HS Token Is Working Correctly

After receiving a new High Security token, follow these steps to verify it is functioning before you need it for an urgent signing task:

1Install the Middleware

Install the token driver/middleware provided by your CA or downloaded from the token manufacturer's website. For HYP2003 HS tokens, install the ePass2003 middleware. For Proxkey WD tokens, install the Proxkey middleware. Run the installer as administrator and restart your PC afterwards.

2Open the Token Management Software

Open the token management application that came with the middleware. The token should appear in the software with its serial number. Verify the serial number matches what is printed on your physical token and confirm it begins with HS (HYP2003) or WD (Proxkey).

3View Your Certificate

In the token management software, view the certificate stored on the token. Confirm the certificate shows your correct name, organisation, and a valid expiry date in the future. If the certificate is not yet loaded (for a freshly issued token), contact your CA.

4Open PDFSign and Confirm Detection

With the token plugged in, open PDFSign. The certificate should appear automatically in the certificate selection dropdown. If it does not appear, click Refresh. Confirm the certificate name matches your DSC details.

5Sign a Test PDF

Open any PDF in PDFSign, place a signature, and click Sign. Enter your PIN when prompted. Verify that the output PDF opens in Adobe Acrobat Reader and shows a valid signature with a green tick. This confirms your High Security token is working correctly end to end.

What If Your HS Token Is Not Working?

If your HS series token is recognised by the token management software but not by PDFSign, or if the signature fails, work through these checks:

  • Driver not installed as administrator: Uninstall and reinstall the middleware using right-click → Run as administrator
  • Wrong middleware version: Some HYP2003 HS tokens require a specific version of the ePass2003 driver. Download the latest version from your CA's support page
  • Token PIN not initialised: A new token from the CA should come with a default PIN on the documentation. If not, contact your CA to initialise the token
  • Certificate not yet loaded: Newly issued tokens sometimes require the CA to complete a final provisioning step. Contact your CA if the certificate is not visible in the token management software
  • USB port issue: Try a different USB port — preferably a USB 2.0 port directly on the PC rather than a USB hub

Frequently Asked Questions

Does my old ePass 2003 token still work after the 2023 mandate?

Yes. If your existing ePass 2003 token was issued before the mandate and carries a valid Class 3 DSC, it continues to work until the DSC expiry date. The mandate applies only to new issuances. When your DSC expires and you renew, your CA will issue the new certificate on a FIPS 140-2 Level 3 High Security token.

How do I know if my token is a High Security token?

Check the serial number on the USB token body. HYP2003 High Security tokens have serial numbers starting with HS. Proxkey High Security tokens have serial numbers starting with WD. You can also look for "FIPS 140-2 Level 3" printed on the token or packaging.

What if I have an HS series token but it is not working?

An HS token that is not working is almost always a driver or middleware issue. Reinstall the ePass2003 middleware as administrator, restart your PC, and try again. If the problem persists, contact your DSC-issuing CA or reach PDFSign support at +91 70574 13999.

Do I need to buy a new token when my DSC expires?

Not necessarily. If your current token is already FIPS 140-2 Level 3 certified, the CA may reload the renewed certificate onto the same hardware. If your token is a pre-mandate Level 2 token, you will receive a new HS token at renewal. Check with your CA during the renewal process.

Does PDFSign work with High Security tokens?

Yes. PDFSign fully supports all PKCS#11-compliant High Security tokens including HYP2003 HS series and Proxkey WD series. Install the correct middleware for your token model and PDFSign detects the certificate automatically — no additional configuration required.
