PDF Digital Signature Not Working? 7 Common Problems Fixed

By PDFSign  ·  May 2026  ·  8 min read
Home › Blog › PDF Digital Signature Not Working

Digital signature issues tend to cluster around a handful of well-known failure points: driver problems, certificate detection, PIN errors, Adobe trust settings, timestamp configuration, document tampering, and batch signing edge cases. Each has a clear fix.

Here are the 7 most common reasons a PDF digital signature stops working — and exactly what to do about each one.

Problem 1 — DSC Token Not Detected

Most Common

Your signing software opens, but no token or certificate is visible. The USB device isn't being recognised by Windows or the application.

Why it happens

  • Token driver not installed or outdated
  • Token inserted into a USB hub or faulty port
  • Wrong driver for the token model (e.g., ePass2003 driver used for a WD-series ProxyKey)
  • Windows hasn't finished enumerating the device (plug-in timing)
Fix:
  1. Unplug the token. Open Device Manager (Win+X → Device Manager). Look for any unknown devices or yellow warning icons under "Smart card readers" or "Other devices".
  2. Download and install the driver specific to your token brand: ePass2003/HYP2003 → SafeNet Authentication Client; ProxyKey/WD series → Watchdata driver; Capricorn → Capricorn middleware.
  3. Plug the token directly into a USB port on the computer chassis — not a hub or extension.
  4. Restart the signing application after inserting the token.

Problem 2 — Certificate Doesn't Appear in the Certificate List

The token is detected (the light is on) but no certificate shows up in the signing software's certificate selector.

Why it happens

  • Application was launched before the token was inserted
  • PKCS#11 middleware not installed or not registered
  • Certificate expired (expired certs are hidden by some applications)
  • The signing software is reading from the Windows Certificate Store instead of the token directly
Fix:
  1. Close the signing application, insert the token, wait 5 seconds, then relaunch the application.
  2. Verify the certificate is present using your token's management software (SafeNet Authentication Client, ProxyKey Manager, etc.). If it's not visible there, the token may need to be re-issued by your CA.
  3. Check the certificate expiry date. If it has expired, you will need a new DSC from your Certificate Authority — an expired cert cannot be renewed, only replaced.
  4. Ensure the PKCS#11 library path is correctly configured in your signing software settings.

Problem 3 — PIN Error or Token Locked

The signing software prompts for a PIN, but the entry is rejected, or the token is locked after multiple failed attempts.

Why it happens

  • Incorrect PIN (factory defaults vary: common defaults are 1234, 12345678, or token-specific)
  • PIN changed by the CA or during issuance and not communicated clearly
  • Too many incorrect attempts triggered the lockout (typically 5–10 failed attempts)
Fix:
  1. Try common default PINs for your token brand. If you received a PIN mailer from your CA, use that PIN.
  2. If locked, open the token's management software. Look for an "Unlock token" or "Change PIN using PUK" option. You'll need the PUK (PIN Unblocking Key) from your CA.
  3. If you don't have the PUK, contact your CA's support. Some CAs can remotely reset the PIN; others require in-person verification.
  4. If the token is permanently locked (usually after 15–20 failed PUK attempts), it must be replaced.
Do not guess the PIN repeatedly. Most tokens lock after 5–10 wrong attempts. If you're unsure, contact your CA before trying.

Problem 4 — "Signature Validity Unknown" in Adobe Acrobat

You open a signed PDF in Adobe Acrobat Reader and see an amber warning: "Signature validity is unknown" or "At least one signature has problems." The document may look perfectly signed but Adobe won't confirm it.

Why it happens

Adobe Acrobat uses its own trusted certificate list called the Adobe Approved Trust List (AATL). Certificate Authorities that are not on the AATL — including several regional CAs — will show as untrusted in Acrobat even if the signature is cryptographically valid.

This is a display/trust configuration issue, not a problem with your signature itself.

Fix (for CCA India root certificates):
  1. Download the CCA India root certificate from the CCA India website.
  2. In Adobe Acrobat Reader: Edit → Preferences → Signatures → Verification → More → Windows Integration → check "Validating Signatures".
  3. Alternatively, import the root cert: Edit → Preferences → Trust Manager → Load Trusted Certificates from Adobe Server (updates AATL) or manually import the downloaded CCA root.
  4. Restart Acrobat and reopen the PDF — the signature should now show as valid.

Problem 5 — RFC 3161 Timestamp Error ("TSA Unreachable")

When you try to sign with a timestamp, the operation fails with an error like "Timestamp server unavailable," "TSA error," or "Connection timed out."

Why it happens

  • Firewall or antivirus is blocking the outbound HTTP request to the TSA server
  • The TSA URL is incorrect in the application settings
  • The TSA service is temporarily down
  • No internet connection is available
Fix:
  1. Verify the TSA URL. Common free TSA servers: http://timestamp.sectigo.com, http://timestamp.digicert.com, http://timestamp.globalsign.com/tsa/r6advanced1.
  2. Check whether your firewall or corporate proxy is blocking outbound HTTP requests to port 80 or 443 for the TSA domain. Temporarily disable the firewall and retry to isolate.
  3. If you're on a restricted network, try a different TSA URL or switch to a network without restrictions.
  4. If internet is not available: sign without a timestamp for now. You can always re-sign with a timestamp once connectivity is restored. A signature without a timestamp is still cryptographically valid — it just lacks a trusted time proof.

Problem 6 — Signature Immediately Shows as Invalid

You sign a PDF, open it, and the signature immediately shows as invalid — even though you just created it seconds ago.

Why it happens

This almost always means the document was modified after signing. The digital signature locks the document content at the moment of signing. Any change — even an invisible metadata update, a PDF viewer saving annotations, or a printer driver modifying the file — will invalidate the signature.

This is the signature working correctly, not a bug.

Fix:
  1. Do not open the signed PDF in any application that might auto-save (some PDF readers add annotations or update metadata on open).
  2. If you need to make any changes to the document, make them before signing, then sign the final version.
  3. If you accidentally modified the signed file, go back to your pre-signature original and sign it again.
  4. Verify by checking the signature panel in your PDF reader — it will show exactly which revision was signed and whether anything changed after.
Note: If a signed PDF is emailed and the recipient's email client converts it (e.g., inline preview, format conversion), this can also invalidate the signature. Always send signed PDFs as plain attachments.

Problem 7 — Batch Signing Fails Mid-Way

You start batch signing a folder of PDFs. The operation signs some files, then stops or skips others without explanation.

Why it happens

  • One or more PDFs in the folder are password-protected (owner or user password)
  • A PDF file is corrupt or not a valid PDF
  • A file is open in another application, causing a file lock conflict
  • A PDF has document restrictions that prevent signing
Fix:
  1. Before batch signing, identify password-protected files. Open each suspected file in Adobe Acrobat Reader — it will prompt for a password if one exists.
  2. Remove passwords from protected PDFs using Adobe Acrobat (if you know the owner password) before including them in the batch.
  3. Close any other applications that may have the PDF files open.
  4. Remove corrupt files: try opening each file individually to confirm it opens cleanly.
  5. If only a few files are problematic, move them out of the batch folder, process the clean files, then handle the problematic ones separately.

Frequently Asked Questions

Why does my digital signature appear valid in one PDF reader but invalid in another?

Different PDF readers maintain different trusted certificate lists. Adobe Acrobat uses the AATL, which does not include some regional Certificate Authorities. If the root certificate of your DSC issuer isn't in the reader's trust store, the signature shows as "Validity Unknown" even though it is cryptographically intact. Fix: add the issuing CA's root certificate to your PDF reader's trust store, or use a reader that already trusts your CA.

Can I add a second digital signature to a PDF that already has one?

Yes. PDFs support multiple signature fields — each signer adds their own signature independently. When a second signer signs, they attest to the document as it existed at the time of their signature. However, modifying the document content (not just adding a new signature) after the first signature will invalidate that first signature. All edits must be complete before the first signer signs.

My DSC certificate has expired — can I still sign new documents?

No. An expired certificate cannot be used to create new valid signatures. You need to obtain a new DSC from your Certificate Authority. Documents you signed while the certificate was valid — and that carry an RFC 3161 timestamp — remain legally valid indefinitely. The timestamp proves the signature was applied when the certificate was still active.
